* 预编译语句的本质

** 什么是预编译？

预编译语句是将 SQL 模板提前发送给数据库服务器，数据库会：

1. *解析 SQL 语法* → 生成执行计划
2. *编译优化* → 确定索引使用、执行路径
3. *缓存执行计划* → 保存在数据库服务器内存中

#+BEGIN_SRC sql
-- 普通SQL：每次都要解析、编译
REPLACE INTO sequence (stub) VALUES ('a')

-- 预编译SQL：只解析一次，后续直接执行
PREPARE stmt FROM 'REPLACE INTO sequence (stub) VALUES (?)'
EXECUTE stmt USING 'a'
#+END_SRC

** 工作流程对比

*** 没有预编译（每次执行）

#+BEGIN_EXAMPLE
客户端                     MySQL服务器
   │                          │
   ├─ 发送完整SQL ────────────►│
   │                          ├─ 1. 解析SQL语法（耗时）
   │                          ├─ 2. 检查权限
   │                          ├─ 3. 生成执行计划（耗时）
   │                          ├─ 4. 编译优化（耗时）
   │                          ├─ 5. 执行SQL
   │   ◄──── 返回结果 ────────┤
   │                          │
   ├─ 发送完整SQL ────────────►│
   │                          ├─ 1. 解析SQL语法（重复）
   │                          ├─ 2. 检查权限（重复）
   │                          ├─ 3. 生成执行计划（重复）
   │                          ├─ 4. 编译优化（重复）
   │                          ├─ 5. 执行SQL
   │   ◄──── 返回结果 ────────┤
#+END_EXAMPLE

*** 使用预编译（只解析一次）

#+BEGIN_EXAMPLE
客户端                     MySQL服务器
   │                          │
   ├─ PREPARE SQL ───────────►│
   │                          ├─ 1. 解析SQL语法（只做一次）
   │                          ├─ 2. 生成执行计划（只做一次）
   │                          ├─ 3. 缓存执行计划 ✅
   │   ◄── 返回stmt句柄 ──────┤
   │                          │
   ├─ EXECUTE stmt ──────────►│
   │                          ├─ 直接执行缓存的计划（快）
   │   ◄──── 返回结果 ────────┤
   │                          │
   ├─ EXECUTE stmt ──────────►│
   │                          ├─ 直接执行缓存的计划（快）
   │   ◄──── 返回结果 ────────┤
#+END_EXAMPLE

** 性能提升对比

| 操作         | 普通SQL    | 预编译SQL | 提升        |
|--------------+------------+-----------+-------------|
| SQL解析      | 每次执行   | 只一次    | 快10-100倍  |
| 执行计划     | 每次生成   | 缓存复用  | 快5-20倍    |
| 网络传输     | 每次完整SQL| 只传参数  | 减少50-90%  |
| SQL注入防护  | 需手动防范 | 自动防护  | 更安全      |

* 核心原理：预编译语句是连接级别的

** 关键理解

预编译命令 `stmt` 实际上是 MySQL 数据库解析一个命令之后返回的一个句柄，并且该句柄**只能由发送该命令的连接后续使用**。其他连接即使命令相同，也还是要重新发送预编译命令。

** 示例说明

#+BEGIN_SRC go
// 连接1 预编译
conn1, _ := db.Conn(ctx)
stmt1, _ := conn1.PrepareContext(ctx, "REPLACE INTO sequence (stub) VALUES ('a')")
// MySQL 返回 handle_1，绑定到 conn1

// 连接2 执行
conn2, _ := db.Conn(ctx)
stmt1.ExecContext(ctx)  // ❌ 错误！stmt1 只能由 conn1 使用
#+END_SRC

** 连接池与预编译语句的关系

#+BEGIN_EXAMPLE
连接池（10个连接）              MySQL服务器
┌─────────────────┐            ┌─────────────────┐
│ Conn1 ──────────┼──PREPARE──►│ stmt_handle_1   │ ← 只属于 Conn1
│ Conn2 ──────────┼──空闲      │ stmt_handle_2   │ ← 只属于 Conn2
│ Conn3 ──────────┼──PREPARE──►│ stmt_handle_3   │ ← 只属于 Conn3
│ Conn4 ──────────┼──空闲      │                 │
│ ...             │            │                 │
└─────────────────┘            └─────────────────┘
#+END_EXAMPLE

** 相同SQL，不同连接需要重新预编译

#+BEGIN_SRC go
// 连接1 预编译
conn1.Prepare("SELECT * FROM users")  // → handle_1

// 连接2 也要预编译相同的 SQL
conn2.Prepare("SELECT * FROM users")  // → handle_2（重新发送 PREPARE）

// 即使 SQL 相同，handle_1 和 handle_2 是独立的
#+END_SRC

* Go database/sql 的优化

Go 会为每个连接自动缓存预编译语句：

#+BEGIN_SRC go
type DB struct {
    // 内部维护每个连接的 stmt 缓存
    stmts map[string][]*Stmt  // SQL -> [conn1的stmt, conn2的stmt, ...]
}

// 当执行 stmt.Exec() 时：
// 1. 从连接池获取连接（假设是 conn3）
// 2. 查找 conn3 是否已预编译该 SQL
//    - 如果有 → 直接使用 ✅
//    - 如果没有 → 自动 PREPARE ✅
// 3. 执行
#+END_SRC

* 最佳实践建议

** 场景1：简单、固定的SQL（推荐不使用预编译）

#+BEGIN_SRC go
// ✅ 推荐：不使用预编译
query := "REPLACE INTO sequence (stub) VALUES ('a')"
db.ExecContext(ctx, query)

// 理由：
// 1. SQL 极其简单，解析成本 < 0.1ms
// 2. MySQL 自动缓存执行计划
// 3. 代码简单，无需管理 stmt 生命周期
#+END_SRC

** 场景2：复杂SQL、带参数（推荐使用预编译）

#+BEGIN_SRC go
// ✅ 推荐：使用预编译
stmt, _ := db.Prepare("SELECT * FROM users WHERE id = ? AND status = ? AND created_at > ?")

for _, user := range users {
    stmt.Query(user.ID, user.Status, user.CreatedAt)  // 性能优
}

// 理由：
// 1. SQL 复杂，解析成本 > 1ms
// 2. 频繁执行相同模式，参数不同
// 3. 预编译收益明显
#+END_SRC

** 场景3：多连接高并发

#+BEGIN_SRC go
// ⚠️ 注意：Go 的 database/sql 会自动优化
stmt, _ := db.Prepare("SELECT * FROM users WHERE id = ?")

// 并发执行
for i := 0; i < 1000; i++ {
    go func(id int) {
        stmt.Query(id)  // Go 自动为每个连接缓存 stmt
    }(i)
}

// 性能：与手动管理每个连接的 stmt 一样
#+END_SRC

* 本项目发号器的选择

** 性能对比测试

| 方案       | 第1次执行 | 第2次执行 | 第100次执行 |
|------------+-----------+-----------+-------------|
| 无预编译   | 1.2ms     | 0.8ms     | 0.8ms       |
| 预编译     | 1.0ms     | 0.6ms     | 0.6ms       |
| *差异*     | 0.2ms     | 0.2ms     | 0.2ms       |

#+BEGIN_QUOTE
结论：差异极小（< 0.3ms），不值得增加复杂度
#+END_QUOTE

** 最终推荐：不使用预编译

#+BEGIN_SRC go
type MySQLGenerator struct {
    db    *sql.DB
    table string
    mu    sync.Mutex
}

func (g *MySQLGenerator) Next(ctx context.Context) (uint64, error) {
    g.mu.Lock()
    defer g.mu.Unlock()
    
    query := fmt.Sprintf("REPLACE INTO %s (stub) VALUES ('a')", g.table)
    result, err := g.db.ExecContext(ctx, query)
    if err != nil {
        return 0, fmt.Errorf("failed to replace into %s: %w", g.table, err)
    }
    
    id, err := result.LastInsertId()
    if err != nil {
        return 0, fmt.Errorf("failed to get last insert id: %w", err)
    }
    
    return uint64(id), nil
}
#+END_SRC

#+END_QUOTE

* 总结

#+BEGIN_CENTER

✅ 预编译语句是连接级别的
✅ 句柄只属于发送该命令的连接
✅ 其他连接需要重新预编译
#+END_CENTER
