* 项目需求背景

在互联网高并发生态中，长链接（如：https://mall.singdile.com/item/goods/electronics/phone?id=837566&source=share_copylink&utm_medium=social&session=xyz...） 存在几大天然的物理劣势，这是催生短链接系统的核心背景：

1. 物理长度限制与传播断层

       痛点：在短信营销、推特（Twitter / X 限制字数）、微博或者手机通知栏里，原生的长链接会瞬间吞噬原本就极度昂贵的字数限制。
    灾难：过长的链接在手机短信里经常会被运营商强制截断或发生物理折行，导致用户点击时链接缺损，直接触发 404，导致高达 30%~50% 的营销流失率。

2. 用户视觉心理防线

    痛点：一长串带有各种诡异参数、Token、乱码的 URL 看起来非常像垃圾诈骗短信、钓鱼网站。

    红利：缩短为类似 t.cn/R3xAx9 这种清爽的、带品牌标识的短链接，能大幅降低用户的心理防线，提升 20%+ 的点击率（CTR）。

3. 数据盲区与精细化运营
    痛点：长链接本身无法承载数据统计。运营团队发出去一千万条短信，根本不知道有多少人点过、在什么时间点、什么手机系统、什么地理位置点过。

    红利：短链接系统作为一个必经的流量中央网关，可以天然、无感地收集全量点击数据，为后方的 BigData  situational awareness（态势感知）和用户画像提供最干净的一手弹药。

* 需求分析
** 核心需求
- 长链接缩短 :: 输入一个合法长链接,返回一个全局唯一的短链接
- 短链接重定向 :: 点击访问短链接，短链接服务器能查询到对应的长链接，并向浏览器发送重定向信号，直接跳转到对应页面
** 未来拓展需求
- 数据分析面板 :: 统计短链接的点击次数等

* 核心需求实现拆解
** 转链模块
- 相同的长链转为同一个短链
- 生成的短链接要尽可能短
** 存储模块
- 保存长链接与短链接的对应关系
- 能根据短链接找到长链接;能根据长链接找到短链接
** 访问链接模块
- 根据短链接查找并重定向长链接的位置
- 后续可能增添访问数据统计

* 总体设计分析
典型的读多写少的系统，数据写入后基本不会改变，可以大胆使用缓存来提高读的效率。
* 短链接生成方式
** 🌐 核心基建背景
在短链接系统的架构设计中，最核心、技术密度最高的环节就是： *如何将一长串 URL 转化为一个全局唯一的短随机码（通常为 6 位字符，如 `6Bv9x1`）* 。
全球可用的 62 进制字符集包括大写字母 `A-Z`、小写字母 `a-z` 以及数字 `0-9`。一个 6 位的短码可以承载的最大容量为  $62^6 \approx 568$ 亿条记录，完全能够吞噬工业海量数据的存储需求。

[[https://tech.meituan.com/2017/04/21/mt-leaf.html][分布式ID生成系统参考]]

** 🚀 流派一：高性能哈希散列算法 (Hash Paradigm)
*** 1. 核心思想
直接将长 URL 经过一个哈希函数变成一个数字，再将数字转化为 62 进制字符串。
- 🔬 算法选型禁区 :: *绝对禁止使用 MD5 或 SHA-256* 。它们不仅计算极其消耗 CPU 算力，且吐出的二进制位数过长（MD5 有 128 位），强行缩短成 6 位会丢失大量信息。
- 👑 工业界标准选择 :: *MurmurHash 算法* 。它是非加密型哈希算法，计算速度比 MD5 快几十倍，且散列分布极度均匀。

*** 2. 💥 核心死穴：哈希碰撞 (Hash Collision)
既然是哈希，就必然存在两个不同的长 URL 算出同一个 6 位短码的概率（哪怕概率只有几百万分之一）。

*** 🛠️ 架构自愈算法 (Collision Resolution Loop)
#+BEGIN_SRC text
[长链接 A 进来] ──► 计算 MurmurHash ──► 得到短码 6Bv9x1
                                              │
                                              ▼
                                 [去数据库/布隆过滤器查验]
                                        /          \
                                       /            \
                                 (不存在)          (不幸碰撞!)
                                     /                \
                                    ▼                  ▼
                              [直接落盘成功]     [战术拼接固定后缀]
                                                (如: URL + "[COLLISION]")
                                                       │
                                                       ▼
                                                [重新触发循环计算]
#+END_SRC

*** 3. 📊 优缺点裁决
- 👍 优点 :: 短链接后缀 *完全随机、没有任何规律* 。竞争对手无法通过递增你的短链接来窃取你公司的总营销流水数据，安全性极佳。
- 👎 缺点 :: 每次写入都必须“先查后写”（先去库里看有没有冲突），在高并发写场景下，对数据库会产生高频的查询轰击内耗。


** 🚀 流派二：分布式全局唯一序列号 (ID Generator Paradigm)
*** 1. 核心思想
*彻底消灭碰撞。* 系统维护一个全局自增的递增计数器。每过来一个长链接，计数器就 `+1`。比如当前递增到了第 `100,500` 个链接，我们就把数字 `100500` 经过数学运算直接转化为 62 进制字符串（即 `000q0U`）。

*** 2. 🧱 三大分布式发号器选型内核
**** A. Redis 原子自增 (~INCR~)
- 物理机制 :: 利用 Redis 单线程的天然特性，一千万个请求进来，顺着 `INCR` 指令排队自增，绝对不会发出重复的数字。
- 架构评价 :: 吞吐量极高（单机可达 10W+ TPS）。但存在单点故障风险，如果 Redis 宕机且持久化不及时，可能会发生发号倒退、导致短码重复的灾难。

**** B. 雪花算法 (~Snowflake~)
- 物理机制 :: 利用时间戳 + 机器 ID + 序列号在内存中高并发揉碎出一个 64 位的分布式唯一 `BIGINT` 数字。
- 架构评价 :: 完全去中心化，本地内存生成，速度快到飞起。但雪花算法算出来的数字通常非常大，转化为 62 进制后，短链接的长度往往会*超过 6 位（通常达到 10 位左右）*，背离了“极致短小”的初衷。
 
**** C. 工业级真神：分布式号段模式 (Segment Mode)
#+CAPTION: 美团 Leaf / 腾讯 Vesta 核心号段批发思想
- 物理机制 :: *“批量倒买倒卖”* 。发号器不再每次去轰击数据库。它每次去 MySQL 里强行申请并锁死一个“号段”（比如拿到 `10001` 到 `20000` 这一万个数字的专有权）。
- 数据流转 :: 发号器把这一万个数字缓存在自己的本地内存里，后方业务过来要号，它在内存里无感 `+1`。等一万个号用光了，它再去数据库里批发下一个号段。

*** 3. 📊 优缺点裁决
- 👍 优点 :: *零冲突，高并发吞吐极其恐怖* 。由于数字是绝对唯一的，写入数据库时不需要执行任何“先查后写”的防御操作，直接盲插（Insert），效率拉满。
- 👎 缺点 :: 短链接是 *严格自增有序的* （如 `000001`, `000002`...）。这会带来严重的商业机密泄露风险：对手只要隔一天分别生成一个短链接，把两者的数字一减，就能肉眼看穿你公司一天的精确短信营销发量。

** 本项目采用的方案
采用发号器的方案，通过Mysql数据库直接生成对应的号.
以MySQL举例，利用给字段设置 =auto_increment_increment和auto_increment_offset来保证ID自增= ，每次业务使用下列SQL读写MySQL得到ID号。
#+begin_src sql
begin;
REPLACE INTO Tickets64 (stub) VALUES ('a');
SELECT LAST_INSERT_ID();
commit;
#+end_src

优点:
- 实现简单，利用已有的数据库功能
- ID自增，满足一些统计要求

缺点:
- 强依赖DB
- ID发号性能瓶颈限制在单台MySQL的读写性能

对于性能缺点，可以使用如下方案解决:
在分布式系统中我们可以多部署几台机器，每台机器设置不同的初始值，且步长和机器数相等。
比如有两台机器。设置步长step为2，TicketServer1的初始值为1（1，3，5，7，9，11...）、TicketServer2的初始值为2（2，4，6，8，10...）。
